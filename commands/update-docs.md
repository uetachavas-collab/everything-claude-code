# ドキュメントを更新する

コードベースと同期したドキュメントを、信頼できる情報源から生成します。

## ステップ 1: 信頼できる情報源を特定する

| 情報源 | 生成するもの |
|--------|-----------|
| `package.json` スクリプト | 利用可能なコマンドのリファレンス |
| `.env.example` | 環境変数のドキュメント |
| `openapi.yaml` / ルートファイル | API エンドポイントリファレンス |
| ソースコードのエクスポート | パブリック API ドキュメント |
| `Dockerfile` / `docker-compose.yml` | インフラセットアップのドキュメント |

## ステップ 2: スクリプトリファレンスを生成する

1. `package.json`（または `Makefile`・`Cargo.toml`・`pyproject.toml`）を読む
2. 説明付きのすべてのスクリプト/コマンドを抽出する
3. リファレンステーブルを生成する：

```markdown
| コマンド | 説明 |
|---------|-------------|
| `npm run dev` | ホットリロード付きで開発サーバーを起動 |
| `npm run build` | 型チェック付きで本番ビルド |
| `npm test` | カバレッジ付きでテストスイートを実行 |
```

## ステップ 3: 環境変数ドキュメントを生成する

1. `.env.example`（または `.env.template`・`.env.sample`）を読む
2. 目的付きのすべての変数を抽出する
3. 必須/任意に分類する
4. 期待されるフォーマットと有効な値を文書化する

```markdown
| 変数 | 必須 | 説明 | 例 |
|----------|----------|-------------|---------|
| `DATABASE_URL` | はい | PostgreSQL 接続文字列 | `postgres://user:pass@host:5432/db` |
| `LOG_LEVEL` | いいえ | ログの詳細レベル（デフォルト: info） | `debug`, `info`, `warn`, `error` |
```

## ステップ 4: コントリビュートガイドを更新する

`docs/CONTRIBUTING.md` を生成または更新する：
- 開発環境のセットアップ（前提条件・インストール手順）
- 利用可能なスクリプトとその目的
- テスト手順（実行方法・新しいテストの書き方）
- コードスタイルの強制（リンター・フォーマッター・pre-commit フック）
- PR 提出チェックリスト

## ステップ 5: ランブックを更新する

`docs/RUNBOOK.md` を生成または更新する：
- デプロイ手順（ステップバイステップ）
- ヘルスチェックエンドポイントと監視
- よくある問題とその修正方法
- ロールバック手順
- アラートとエスカレーションパス

## ステップ 6: 鮮度チェック

1. 90 日以上更新されていないドキュメントファイルを見つける
2. 最近のソースコードの変更と照合する
3. 手動レビューが必要な古くなった可能性のあるドキュメントにフラグを立てる

## ステップ 7: サマリーを表示する

```
ドキュメント更新
──────────────────────────────
更新済み:  docs/CONTRIBUTING.md（スクリプトテーブル）
更新済み:  docs/ENV.md（新変数 3 件）
要確認:  docs/DEPLOY.md（142 日前から古い）
スキップ:  docs/API.md（変更なし）
──────────────────────────────
```

## ルール

- **単一の信頼できる情報源**: 常にコードから生成し、生成されたセクションは手動で編集しない
- **手動セクションを保持する**: 生成されたセクションのみを更新し、手書きの文章はそのままにする
- **生成コンテンツにマークをつける**: 生成されたセクションの周りに `<!-- AUTO-GENERATED -->` マーカーを使用する
- **求められていない限りドキュメントを作成しない**: コマンドが明示的に要求する場合のみ新しいドキュメントファイルを作成する
