# 戦略的コンパクトスキル

任意の自動コンパクションに頼るのではなく、ワークフローの戦略的なポイントで手動 `/compact` を提案します。

## 有効化するとき

- コンテキスト上限（20 万トークン以上）に近づいている長いセッション
- 複数フェーズのタスク（調査 → 計画 → 実装 → テスト）
- 同じセッション内の無関係なタスク間の切り替え
- 主要なマイルストーンを完了して新しい作業を始めるとき
- 応答が遅くなったり一貫性がなくなったりするとき（コンテキストの圧力）

## なぜ戦略的コンパクションか?

自動コンパクションは任意のタイミングで発生する：
- タスクの途中で重要なコンテキストが失われることがある
- 論理的なタスク境界を認識しない
- 複雑なマルチステップ処理を中断する可能性がある

論理的な境界での戦略的コンパクション：
- **探索後・実行前** — 調査コンテキストをコンパクトにし、実装計画を保持する
- **マイルストーン完了後** — 次のフェーズのためにフレッシュスタートする
- **主要なコンテキスト切り替え前** — 別のタスクの前に探索コンテキストをクリアする

## 仕組み

`suggest-compact.js` スクリプトが PreToolUse（Edit/Write）で実行され：

1. **ツール呼び出しを追跡** — セッションのツール呼び出し回数をカウント
2. **閾値検出** — 設定可能な閾値（デフォルト: 50 回）で提案
3. **定期リマインダー** — 閾値後 25 回ごとにリマインド

## コンパクション決定ガイド

| フェーズ移行 | コンパクト? | 理由 |
|-----------------|----------|-----|
| 調査 → 計画 | はい | 調査コンテキストは大きい。計画が蒸留されたアウトプット |
| 計画 → 実装 | はい | 計画は TodoWrite またはファイルに保存済み。コード用にコンテキストを解放 |
| 実装 → テスト | 場合による | テストが最近のコードを参照する場合は保持。焦点が変わる場合はコンパクト |
| デバッグ → 次の機能 | はい | デバッグトレースは無関係な作業のコンテキストを汚染する |
| 実装途中 | いいえ | 変数名・ファイルパス・部分的な状態を失うのはコスト高 |
| 失敗したアプローチの後 | はい | 新しいアプローチを試みる前に行き詰まった推論をクリアする |

## コンパクション後に残るもの

| 残る | 失う |
|----------|------|
| CLAUDE.md の指示 | 中間の推論と分析 |
| TodoWrite のタスクリスト | 以前読んだファイルの内容 |
| メモリファイル（`~/.claude/memory/`） | マルチステップの会話コンテキスト |
| Git の状態（コミット・ブランチ） | ツール呼び出しの履歴と回数 |
| ディスク上のファイル | 口頭で述べたニュアンスのあるユーザー設定 |

## ベストプラクティス

1. **計画後にコンパクト** — 計画が TodoWrite に確定したら、フレッシュスタートのためにコンパクトする
2. **デバッグ後にコンパクト** — 作業を続ける前にエラー解決コンテキストをクリアする
3. **実装途中はコンパクトしない** — 関連する変更のためにコンテキストを保持する
4. **提案を読む** — フックがいつかを教えてくれる。するかどうかはユーザーが決める
5. **コンパクト前に書き込む** — コンパクト前に重要なコンテキストをファイルまたはメモリに保存する
6. **サマリー付きで `/compact`** — カスタムメッセージを追加する: `/compact 次に auth ミドルウェアの実装に集中する`

## 関連

- メモリ永続化フック — コンパクションを超えて状態を維持するため
